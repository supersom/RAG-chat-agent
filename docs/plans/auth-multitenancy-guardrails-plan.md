# Auth, Multi-Tenancy & Guardrails for customer-support-agent

> Original implementation plan, preserved here for reference. Written before implementation began, approved, then executed via 10 sequenced tasks (each with an independent implementer + reviewer pass) plus a final whole-branch review. See [`../auth-multitenancy-guardrails.md`](../auth-multitenancy-guardrails.md) for what actually shipped, how it diverged from this plan, and the AWS resources provisioned for it.

## Context

The app today is fully stateless: no auth, no DB, no middleware. Every "setting" (LLM API key, AWS creds, knowledge base ID) lives in browser `localStorage` and is sent as plain fields in the `/api/chat` POST body — the server trusts whatever the client sends. Anyone can call `/api/chat`/`/api/logs` with arbitrary credentials, there's no multi-tenant isolation, and no content moderation. There's also a live bug that becomes a real cross-tenant leak once multiple tenants share the process: `app/api/chat/route.ts` mutates `process.env.OPENAI_API_KEY` globally per-request instead of passing the key directly to the completion call.

## Decisions (confirmed, do not re-litigate)

1. **Auth**: NextAuth.js / Auth.js **v5** (`next-auth@beta`), Credentials provider, **JWT session strategy** (required — Credentials provider can't use database sessions). Two roles: `admin`, `end_user`.
2. **Who logs in**: both admins and end-users authenticate, but each **tenant** has a `requireEndUserAuth` toggle for anonymous/guest chat. Admin login is always required for the config surface.
3. **Tenant identification**: a **signed JWT "tenant embed token"** (distinct from the NextAuth session JWT), generated by an admin, sent as an `x-tenant-token` header, verified on every tenant-scoped request.
4. **Guardrails (this phase only)**: AWS Bedrock Guardrails via `ApplyGuardrailCommand` on input and output text, regardless of which LLM generated it. Provider-agnostic custom guardrails (rate limiting, etc.) are **backlogged**.
5. **Per-tenant secrets**: for this phase, all tenants share the deployment's existing server-side AWS creds (`BAWS_*` env vars) and LLM provider keys (`OPENAI_API_KEY`/`ANTHROPIC_API_KEY`/`OPENROUTER_API_KEY` env vars) — no raw secrets are stored in DynamoDB. `Tenant.awsCredentialsSecretArn` exists in the schema for forward-compat but its resolution (Secrets Manager lookup) is **not implemented this phase**; if set, log a warning and fall back to the shared env creds.

## Architecture: where tenant resolution happens

**Inside API route handlers (Node runtime), not `middleware.ts`.** DynamoDB reads and bcrypt want Node; the app has zero Edge investment today; Amplify SSR is regional Lambda either way. `middleware.ts` only gates `/admin/*` **page** navigation via NextAuth's edge-safe JWT-cookie check (`matcher: ["/admin/:path*"]` — must NOT match `/api/chat`, `/api/logs`, `/api/auth/*`). All real authorization (tenant token verify + DynamoDB lookup + optional end-user session check) happens in one chokepoint, `resolveTenantContext()`, called from inside each tenant-scoped route.

## Global Constraints (verbatim values — hand to every reviewer)

- Tenant embed token header: **`x-tenant-token`**. JWT library: **`jose`**, HS256, secret from `process.env.TENANT_JWT_SECRET`, claim `aud: "tenant-embed"` (this exact string), payload `{ tenantId, allowedOrigins? }`.
- NextAuth session JWT: `session.strategy: "jwt"`, custom claims `role: "admin" | "end_user"` and `tenantId: string` added via `callbacks.jwt`/`callbacks.session`. Secret: `process.env.AUTH_SECRET`.
- HTTP status conventions across all new/modified routes: **401** = missing/invalid/expired token or missing required session; **403** = valid session but wrong role/tenant; **400** = request body fails zod validation; **404** = tenant/resource not found.
- Roles are exactly the strings `"admin"` and `"end_user"` (snake_case), everywhere — no other casing.
- DynamoDB tables: `process.env.DYNAMODB_TENANTS_TABLE`, `process.env.DYNAMODB_USERS_TABLE` (no hardcoded table names in code).
- Users table: email uniqueness is scoped to `(tenantId, email)`, **not global**. Admin login is the deliberate exception — looked up by email alone (`getUserByEmailAnyTenant`) since tenant isn't known until after auth. Comment this asymmetry in code where it occurs.
- Never store raw AWS credentials or raw LLM API keys in DynamoDB. Never trust client-supplied credentials/secrets in `/api/chat` or `/api/logs` request bodies — those fields must not exist in the accepted request shape at all.
- Client-side dev-mode gating must use a **build-time** `NEXT_PUBLIC_APP_ENV` env var, never a client-supplied/localStorage flag, for anything security-relevant. Server-side dev-mode fallback (in `resolveTenantContext`) checks the real `process.env.NODE_ENV`, never a client flag.
- Guardrail failure handling: **INPUT** screening fails **closed** (block/degrade the request on any error calling the guardrail). **OUTPUT** screening fails **open** with logging (don't block a response that already passed generation just because the guardrail call itself errored).
- No live AWS resources (DynamoDB tables, Bedrock Guardrail) exist yet in any account — they are provisioned manually, outside this task sequence. **No task in this plan can perform a live end-to-end test against real AWS.** Verification per task is: `npm run lint`, `npm run typecheck`, and (Task 10 onward) `npm run test` against mocked AWS clients. Do not attempt to run `npm run dev` and exercise a real signup/chat flow — it will fail for lack of provisioned infrastructure, and that is expected, not a bug to fix.

## Data layer schema (DynamoDB)

- **`Tenants`**: PK `tenantId` (string, ULID). Fields: `name`, `knowledgeBaseId`, `llmProviderDefaults: { provider: string; model: string; allowedModels?: string[] }`, `requireEndUserAuth: boolean`, `guardrailId: string`, `guardrailVersion: string`, `allowedOrigins?: string[]`, `awsCredentialsSecretArn?: string`, `createdAt: string (ISO)`.
- **`Users`**: PK `userId` (string, ULID). GSI `email-index`: PK `email`, SK `tenantId`. Fields: `email`, `passwordHash` (bcryptjs), `role: "admin" | "end_user"`, `tenantId`, `createdAt: string (ISO)`.

## Task 4: Tenant embed token (jose) + resolveTenantContext + embed-token issuance route

**Depends on:** Task 1 (`getTenant`), Task 2 (`auth()`).

**Files:**
- New `app/lib/tenant.ts`:
  - `export interface TenantEmbedTokenClaims { tenantId: string; allowedOrigins?: string[]; }`
  - `signTenantToken(claims: TenantEmbedTokenClaims, options?: { expiresIn?: string }): Promise<string>` — `jose`'s `SignJWT`, `.setProtectedHeader({ alg: "HS256" })`, `.setIssuedAt()`, `.setAudience("tenant-embed")`, conditionally `.setExpirationTime(options.expiresIn)` if provided, `.sign(new TextEncoder().encode(process.env.TENANT_JWT_SECRET))`.
  - `verifyTenantToken(token: string): Promise<TenantEmbedTokenClaims | null>` — `jose`'s `jwtVerify(token, secret, { audience: "tenant-embed" })`; catch any error and return `null` (never throw to the caller).
  - `export type TenantContext = Tenant & { awsCredentials: { accessKeyId: string; secretAccessKey: string } };` (import `Tenant` from `app/lib/db/schema.ts`).
  - `export type TenantResolutionError = { error: string; status: number };`
  - `export function isTenantResolutionError(x: TenantContext | TenantResolutionError): x is TenantResolutionError` — type guard checking for the `status` field.
  - `resolveTenantContext(req: Request): Promise<TenantContext | TenantResolutionError>`:
    1. `const token = req.headers.get("x-tenant-token");`
    2. If no token: if `process.env.NODE_ENV === "development"`, return a synthesized dev `TenantContext` (`tenantId: "dev"`, `knowledgeBaseId: process.env.KNOWLEDGE_BASE_ID ?? ""`, `llmProviderDefaults: { provider: "anthropic", model: process.env.DEFAULT_MODEL ?? "claude-3-5-sonnet-20240620" }`, `requireEndUserAuth: false`, `guardrailId: process.env.BEDROCK_GUARDRAIL_ID ?? ""`, `guardrailVersion: process.env.BEDROCK_GUARDRAIL_VERSION ?? "DRAFT"`, `createdAt: new Date().toISOString()`, `awsCredentials: { accessKeyId: process.env.BAWS_ACCESS_KEY_ID!, secretAccessKey: process.env.BAWS_SECRET_ACCESS_KEY! }`); else return `{ error: "Missing tenant token", status: 401 }`.
    3. `const claims = await verifyTenantToken(token);` — if `null`, return `{ error: "Invalid or expired tenant token", status: 401 }`.
    4. If `claims.allowedOrigins?.length`, compare against `req.headers.get("origin")`; if the origin header is present and not in the list, return `{ error: "Origin not allowed", status: 403 }` (if no origin header at all, allow — server-to-server/non-browser calls have no origin).
    5. `const tenant = await getTenant(claims.tenantId);` — if `null`, return `{ error: "Unknown tenant", status: 404 }`.
    6. If `tenant.requireEndUserAuth`: `const session = await auth();` (import from `@/auth`) — if `!session || session.user.tenantId !== tenant.tenantId`, return `{ error: "Authentication required", status: 401 }`.
    7. Resolve `awsCredentials`: if `tenant.awsCredentialsSecretArn` is set, `console.warn("Per-tenant AWS credential resolution not implemented this phase, falling back to shared credentials", tenant.tenantId)` and still use the shared env creds below (do not implement Secrets Manager lookup).
    8. Return `{ ...tenant, awsCredentials: { accessKeyId: process.env.BAWS_ACCESS_KEY_ID!, secretAccessKey: process.env.BAWS_SECRET_ACCESS_KEY! } }`.
- New `app/api/tenant/embed-token/route.ts` — `POST`: `const session = await auth(); if (!session || session.user.role !== "admin") return Response.json({ error: "Forbidden" }, { status: 403 });`, zod-validate body `{ allowedOrigins?: string[], expiresIn?: string }`, call `signTenantToken({ tenantId: session.user.tenantId, allowedOrigins }, { expiresIn })`, return `{ token }`.
